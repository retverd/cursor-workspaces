# Архитектура правил Cursor

## Назначение

Этот документ описывает концепцию шаблонного репозитория правил для Cursor.
Его цель — зафиксировать архитектуру контекста, чтобы дальше развивать шаблон уже внутри самого Cursor, а не пересобирать логику с нуля в каждом новом проекте.

## Основная идея

Вместо одного большого управляющего файла используется набор небольших правил и документов, разделённых по ответственности. Такой подход делает систему более переиспользуемой, масштабируемой и устойчивой к росту проекта.

Главная идея:

- глобальные правила задают поведение агента и устойчивые инварианты;
- локальные правила задают ограничения для отдельных компонентов и технологий;
- документация в `docs` служит источником концептуального и архитектурного контекста;
- новые правила появляются только под повторяющиеся типовые ошибки, а не под единичные задачи.

## Слои контекста

### 1. Слой поведения агента

Это универсальные правила поведения модели. Они должны быть короткими, стабильными и пригодными для большинства проектов.

Примеры:

- `00-agent-core.mdc`
- `90-verification.mdc`

Назначение слоя:

- уменьшать галлюцинации;
- удерживать фокус на минимально достаточном решении;
- требовать явной проверки результата;
- формировать предсказуемое поведение агента.

### 2. Слой концепции проекта

Это правила и документы, которые описывают цели проекта, архитектурные принципы, границы компонентов и общие подходы. Они не должны быть жёстко привязаны к конкретному стеку технологий.

Примеры:

- `02-architecture-principles.mdc`
- `docs/architecture/overview.md`
- `docs/architecture/rules.md`

Назначение слоя:

- сохранять концептуальную целостность системы;
- позволять менять технологии без разрушения общей идеи;
- давать агенту устойчивую ментальную модель проекта.

### 3. Слой платформы и компонентов

Это правила для языков, фреймворков, подсистем и отдельных компонентов.
Они должны быть локальными и подключаться только там, где действительно нужны.

Примеры:

- `1x-frontend-react.mdc`
- `1x-backend-api.mdc`
- `1x-python-data.mdc`
- будущие `1x-*` правила для конкретных компонентов и технологий

Назначение слоя:

- давать технические ограничения локально;
- не загрязнять глобальный контекст;
- снижать типичные ошибки в конкретной зоне кода.

### 4. Слой документации

Это файлы в `docs`, которые помогают людям и агентам понимать проект без чтения истории чатов.

Назначение слоя:

- сохранять источник истины по концепции и решениям;
- отделять архитектурные идеи от операционных инструкций правил;
- помогать наращивать контекст проекта эволюционно.

## Принципы проектирования правил

### 1. Одна ответственность на файл

Каждый файл правил должен решать одну основную задачу.
Если в правило попало слишком много разных тем, его нужно разделить.

### 2. Короткие глобальные правила

Глобальные правила должны быть короткими и высокосигнальными.
Постоянно подключённый контекст дорогой, поэтому в него должны попадать только устойчивые инструкции.

### 3. Локальные технические правила

Технические детали должны жить в ограниченных правилах через `globs`.
Это уменьшает шум и делает правила более точными.

### 4. Правила для действий, документация для объяснения

Правила нужны для действий агента.
Документация нужна для объяснения целей, ограничений, контекста и решений.
Не нужно копировать одно в другое.

### 5. Развивать из повторяющихся ошибок

Новые правила создаются не потому, что "так может пригодиться", а потому что возник повторяющийся паттерн ошибок, расхождений или некачественных решений.

## Роли файлов

### Глобальные правила

- `00-agent-core.mdc` — базовая дисциплина работы агента;
- `01-rules-generation.mdc` — как развивать саму систему правил;
- `02-architecture-principles.mdc` — принципы архитектуры;
- `03-repo-operations.mdc` — правила для изменений репозитория;
- `90-verification.mdc` — финальная проверка результата.

### Ограниченные правила

- `04-docs-concepts.mdc` — правила генерации документации в `docs`;
- `1x-*` — ограниченные правила для отдельных компонентов и технологий.

## Концепция генерации ограниченных правил (1X)

Префикс `1x-` используется для правил отдельных компонентов, слоёв, технологий и повторяющихся локальных процессов.
Это может быть:

- `1x-frontend-react.mdc`
- `1x-backend-api.mdc`
- `1x-data-pipelines.mdc`
- `1x-testing-typescript.mdc`
- `1x-docs-adr.mdc`

Такие правила:

- не должны быть `alwaysApply` по умолчанию;
- должны иметь понятные `globs`;
- должны описывать локальные ограничения, типичные риски и антипаттерны;
- должны появляться только когда локальный контекст действительно даёт выигрыш.

## Структура репозитория

Рекомендуемая структура шаблонного репозитория:

```text
.cursor/
  rules/
    00-agent-core.mdc
    01-rules-generation.mdc
    02-architecture-principles.mdc
    03-repo-operations.mdc
    04-docs-concepts.mdc
    90-verification.mdc
    1x-*.mdc
    8x-*.mdc

docs/
  architecture/
    overview.md
    decisions.md
    rules.md
  components/
    *.md
```

## Стратегия изменений

Изменение шаблона должно происходить аккуратно:

1. Сначала фиксируются наблюдаемые повторяющиеся проблемы.
2. Потом определяется, это проблема правил, документации или структуры проекта.
3. Затем вносятся минимальные изменения.
4. После этого оценивается, уменьшилось ли число повторяющихся проблем.

## Чем этот репозиторий не является

Этот репозиторий:

- не является конкретной реализацией одного проекта;
- не должен содержать стек-специфику всех будущих проектов;
- не должен превращаться в длинную энциклопедию пожеланий для модели.

Его задача — быть устойчивым каркасом для запуска новых проектов и дальнейшего эволюционного развития правил уже изнутри Cursor.

## Следующий путь развития

Дальнейшее развитие можно вести прямо в Cursor:

1. использовать `01-rules-generation.mdc` для создания новых ограниченных правил;
2. развивать `docs/architecture/rules.md` как описание операционной модели;
3. адаптировать `1x-*` правила под конкретные типы проектов;
4. держать глобальные правила короткими и стабильными;
5. пересматривать структуру только при появлении системных проблем.
