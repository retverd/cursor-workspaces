# Cursor Template Repository — Суммаризация диалога

## Отправная точка

Отправной точкой стал файл `CLAUDE.md` из репозитория Андрея Карпати — образцовый пример того, как структурировать инструкции для AI-ассистента в IDE. На его основе были разработаны базовые правила поведения агента в Cursor, а затем вся концепция выросла в отдельный template-репозиторий для управления контекстом.

## Ключевая идея: Context Engineering

Вместо одного большого файла инструкций вся система строится как набор небольших, разделённых по ответственности правил и документов. Глобальные правила задают поведение агента и устойчивые инварианты. Scoped rules через `globs` задают локальные ограничения для отдельных компонентов. Документация в `docs` служит source of truth для концепции и архитектурных решений. Новые правила появляются только под повторяющиеся failure modes, а не под единичные ситуации.

## Архитектура правил

Система делится на четыре слоя контекста:

| Слой | Файлы | Назначение |
|---|---|---|
| Agent Behavior | `00-agent-core.mdc`, `90-verification.mdc` | Поведение агента, проверка результата |
| Project Concept | `02-architecture-principles.mdc`, `docs/architecture/` | Цели и архитектурные принципы |
| Platform & Components | `---20-platform-common.mdc`, `1x-*.mdc.example` | Техстек, компоненты, локальные правила |
| Documentation | `04-docs-concepts.mdc`, `docs/**` | Source of truth для целей и решений |

## Созданные файлы

### Core global rules (`alwaysApply: true`)
- `00-agent-core.mdc` — дисциплина работы агента: формулировать задачу, не скрывать неопределённость, делать минимально достаточное изменение, self-review
- `01-rules-generation.mdc` — meta-rule: когда и как создавать новые rules, типы правил, шаблон для `3x-*`, антипаттерны
- `02-architecture-principles.mdc` — архитектурные принципы
- `90-verification.mdc` — definition of done: список изменённых файлов, обоснование, риски, тесты

### Governance rules
- `---20-platform-common.mdc` — общие техограничения платформы
- `03-repo-operations.mdc` — правила для рискованных изменений репозитория

### Scoped rules (`alwaysApply: false`, с `globs`)
- `04-docs-concepts.mdc` — правила генерации и поддержки документации в `docs`
- `1x-frontend-react.mdc.example` — scoped rule для TypeScript/React фронтенда
- `1x-backend-api.mdc.example` — scoped rule для API-сервисов
- `1x-python-data.mdc.example` — scoped rule для Python и data pipelines

### Архитектурные документы (`docs/`)
- `docs/architecture/overview.md` — шаблон для описания системы на technology-neutral уровне
- `docs/architecture/decisions.md` — лёгкий ADR-лог архитектурных решений
- `docs/architecture/rules.md` — полная концепция шаблонного репозитория: слои, роли файлов, naming, operating model, стратегия изменений
- `docs/components/frontend.md`, `backend.md`, `data-platform.md` — компонентный контекст

### Workflow-документы (`docs/workflows/`)
- `new-project-bootstrap.md` — как инициализировать новый проект из шаблона
- `adding-new-rule.md` — как добавлять новые rules
- `rule-review.md` — процесс ревью: когда нужен ревью, вопросы, outcomes, checklist
- `evaluation.md` — evaluation loop: failure mode taxonomy, тестовые задачи, scoring, критерии успеха

### Шаблоны (`docs/templates/`)
- `component-doc-template.md` — шаблон документа компонента: Purpose, Scope, Invariants, Responsibilities, Constraints, Contracts, Failure Modes
- `adr-template.md` — шаблон ADR: Context, Decision, Alternatives, Consequences, Migration
- `rule-change-proposal.md` — proposal для изменения rules: problem, proposed change, scope classification, expected effect, evaluation plan, sunset condition

### Infrastructure
- `scripts/bootstrap.sh` — скрипт инициализации нового проекта из шаблона
- `CHANGELOG.md` — версионирование шаблона (PATCH/MINOR/MAJOR)
- `ROADMAP.md` — milestones v1.0: baseline stabilization → governance → evaluation loop → release
- `.gitignore` — commit `.cursor/` и `docs/`, игнорировать только локальный шум
- `.cursor/README.md` — описание naming conventions для rules

## Принципы проектирования

**1. One responsibility per file.** Каждый файл rules решает одну основную задачу. Если в правиле смешалось несколько тем — его нужно разделить.

**2. Short global rules.** AlwaysApply-контекст дорогой. В него должны попадать только устойчивые, высокосигнальные инструкции.

**3. Scoped technical rules.** Технические детали через `globs`. Это уменьшает шум и делает правила более точными.

**4. Rules are operational, docs are explanatory.** Rules нужны для действий агента. Docs нужны для объяснения целей, ограничений, контекста и решений. Не нужно копировать одно в другое.

**5. Evolve from repeated mistakes.** Новые rules создаются только под повторяющийся паттерн ошибок, а не "на всякий случай".

## Naming conventions

| Префикс | Тип | Описание |
|---|---|---|
| `00–09` | Global governance | Поведение агента, meta-rules, архитектурные принципы |
| `1x-*` | Component/scoped | Локальные правила для компонентов и технологий |
| `80–89` / `8x-*` | Temporary/migration | Временные и переходные правила |
| `90+` | Verification | Проверка и reporting |

## Process развития шаблона

1. Зафиксировать observed failure mode.
2. Решить — проблема в rules, docs или структуре.
3. Внести минимальное изменение.
4. Прогнать те же кейсы повторно.
5. Сохранить изменение только если оно реально улучшило результат без роста контекстного шума.

## Рекомендации для продолжения в Cursor

1. **Перенести все core rules** из этого диалога в `.cursor/rules/` нового template-репозитория.
2. **Заполнить** `docs/architecture/overview.md` и `docs/components/*.md` под реальный проект.
3. **Проверить `globs`** под реальную структуру репозитория (monorepo, `apps/*`, `services/*`, `packages/*`).
4. **Запустить** `scripts/bootstrap.sh` на пустом репозитории, убедиться что workflow работает.
5. **Собрать первый eval-набор** из 8–12 реальных задач: bugfix, API-contract change, docs update, migration.
6. **Развивать шаблон** через `01-rules-generation.mdc` и `rule-change-proposal.md` — только под повторяющиеся ошибки.
7. **v1.0 release gate:** core rules выровнены по стилю → нет дублей между README и docs → bootstrap проверен → examples согласованы → changelog обновлён.

## Итоговая оценка

Созданная система — это не просто набор Cursor-настроек, а framework для управления контекстом агента. Разделение поведения, концепта, техслоя, документации, examples и bootstrap-механики позволяет масштабировать шаблон без превращения его в хаотичную свалку правил. Главный риск на этапе v1.0 — не нехватка правил, а преждевременное расширение вширь. Поэтому приоритет — стабильный минималистичный baseline и дисциплинированный evaluation loop.
